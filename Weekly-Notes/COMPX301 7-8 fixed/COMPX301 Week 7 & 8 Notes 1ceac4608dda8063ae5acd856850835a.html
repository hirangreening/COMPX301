Regular Expressions cont.

Some important terms to know:

Expression (E) = start state of the machine (and of the context free grammar)
Term (T)
Factor (F)
Vocab (V) = A literal

Precedence of regex operations:

() brackets -> * closure -> + alternation

Rules:

E -> T   // An expression can be a term
E -> TE  // Or it can be a term followed by another expression (Concatenation)

T -> F   // A term can be a factor
T -> F*  // A term can be a factor 0 or more times (Closure)
T -> F+F // A term can be one of two factors (Alternation)

F -> V   // A factor can be a literal
F -> (E) // A factor can be a whole expression enclosed in brackets

Doing this is called bottom-up parsing, or data-driven parsing.

We are going to build a top-down, depth first parser. It’s called a Left-Recursive Descent Parser.

There are also:

  • Width-first parsing expands each possible node at each level.
  • Depth-first parsing expands fully down one path first, then backtracks if required.

We want this parser to build the corresponding finite state machine. We can build the machine while the parser is running. This turns our parser into a FSM compiler.

To do this we need several arrays and a global integer value called “state” which starts at 1 and represents the next state to be built.

Parser modified to start building our compiler.

main() {
  str p = "a(b+c)*d";   // The regex to parse
  int state = 1;
  int j = 0;
  setState            // j is the index to where we are looking in the string

  expression();
}

int expression() {
  int r;
  r = term();            // Return the start state of the next machine
  if (j == p.length - 1) 
	  return;              // No more input
	if (isVocab(p[j]) || p[j] == '(')
    expression();        // We are looking at another expression (concat)
  else if (j == p.length - 1)
    setState(state, '!', -1, -1); // Set the final state.
  else 
    error();
  return r;
}

int term() {
  int f = factor();  
  if (p[j] != '*' && p[j] != '+')
	  return f;
	else if (p[j] == '*') // We see a closure
	  setState(state, 'BR', f, state + 1) 
	  state++;
	  j++;
	  return state-1;
  else if (p[j] == '+') // We see + which means "one or more".
    j++;                   
    factor();      
	  return;
  return;                // T -> F
}

int factor() {
  if (isVocab(p[j]))     // Check if the current character is a valid vocab character
    setState(state, p[j], state + 1, state + 1);
    state++;
    j++;                 // F -> V
    return state - 1;
  if (p[j] == '(')       // We are seeing another expression
    int r;               // this is going to be our start of the next expression
    j++;                 // F -> (E)
    r = expression();
    if (p[j] != ')')     // The expression did not return with a closed bracket
      error();           // so it's an error
    j++;
    return r;            // Return the start of the next state
  else
    error();             // factor must be a literal or a '(', if not its an error
}

Important things to consider for the assignment:

The order of precedence:

  • escaped characters (i.e. symbols preceded by \)
  • parentheses (i.e. the most deeply nested regexps have the highest precedence)
  • repetition/option operators (i.e. * and ? and +)
  • concatenation
  • alternation (i.e. | )

Escaped characters:

Anything preceded by a “\” character will be treated as a literal. This relates to characters that usually have some meaning in the regex such as: * ? + ( ) \ and .

Wildcard symbol:

the period “.” is a wildcard symbol that can match any literal (single character).

Parentheses:

Anything within parentheses should be treated as its own expression and should be processed first before moving on.

Repetition/option operators (* ? +):

The order at which you process these shouldn’t really matter too much. Your assignment won’t be tested on badly formed expressions like “ab+?c”.

What these mean (as per assignment outline):

  • * zero or more occurrences of the previous FSM
  • ? zero or one occurrences of the previous FSM
  • + one or more occurrences of the previous FSM

Concatenation:

Joining two smaller regular expression together i.e “abc”.

Alternation:

The “or” statement is applied last.

Concatenation/Alternation (important).

Be careful about the precedence here as this isn’t what was described in the lectures.

Because concatenation is done before alternation is means that if you gave a regex such as “abc|def” to your program it should be treated as “(abc)|(def)”. This differs from how it was described in lectures which would treat it as: “ab(c|d)ef”. So be careful with this!

Making your life easier/edge cases:

I would recommend adding a final branching state to various operations within your regex, and I will explain why:

For the | operator (using the regex a(b|c)d):

state num12345678
char chBRabBRcBRdBR
int next12463678-1
int next22465678-1

Notice that there is an extra branching state where the literals effected by the | operator point to (state 6).

This can make it easier to handle some certain cases such as “a((b|c)|d)”. One possible set of states for this regex is listed below:

Without adding the extra BR state after

state num12345678
char chBRabBRcBRdBR
int next12683848-1
int next22685878-1

Note that above without the extra BR state you are forced to update where states 3, 5 and 2 are pointing to (although this is not the only way to handle this). This can get very messy when there are a lot of nested parentheses and sequential alternations.

With the extra BR state:

state num12345678910
char chBRabBRcBRBRdBRBR
int next12763694910-1
int next22765698910-1

As shown above: The only previous things we needed to go back and update were state 2 and state 6 to point to the new BR states.

You can put these additional BR states onto the end of any expression that uses a branching state if you find it easier (* + ?)

Keep in mind that there are several ways to do this, none of which are “wrong” if they work.

Make sure you manually validate the following regex to make sure that your program is handling them correctly:

  • ((a|b)|c)
  • ((abc|def)|hij)
  • a*b*
  • a+b+
  • a?b?
  • Combinations of the above such as “(a|b)?c*” etc.

function LIS(arr):
  n = length of arr
  L = array of size n, initialized to 1

  for i from 1 to n-1:
    for j from 0 to i-1:
      if arr[i] > arr[j]:
        L[i] = max(dp[i], L[j] + 1)

  return max value in dp

Calculating the LCS

Here is some pseudocode for a dynamic programming problem similar to what was described in Fridays lecture (in the sense that it uses a 2d array). This will determine what the length of the longest common subsequence is (although it won’t actually find that sequence). But it can be used to find the minimum edit distance using the equation above.

function LCS(X, Y):
  m = length of X
  n = length of Y
  L = 2D array of size (m+1) x (n+1), initialized to 0

  for i from 1 to m:
	  for j from 1 to n:
      if X[i-1] == Y[j-1]:
        L[i][j] = L[i-1][j-1] + 1
      else:
        L[i][j] = max(dp[i-1][j], L[i][j-1])

  return dp[m][n]
